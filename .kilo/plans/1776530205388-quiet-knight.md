# Paper Framework Usage Audit

## Objective
Review how the editor utilizes Paper framework compared to best practices for Unity-like editors.

## Summary

| Issue | Severity | Description |
|-------|----------|-------------|
| Missing Virtualization | HIGH | No VirtualList for large entity/component lists |
| Direct State Mutation | MEDIUM | Mutating entity properties directly instead of controlled |
| AlwaysRender = true | MEDIUM | Unnecessary rendering every frame |
| Missing Memoization | MEDIUM | Repeated expensive reflection calls per frame |
| No Key Props | LOW | Missing keys in list mappings |

---

## HIGH Priority Issues

### 1. Missing Virtualization (HierarchyPanel.csx, InspectorPanel.csx, AssetBrowserPanel.csx)

**Issue**: All entities/components rendered as individual UI nodes - no virtualization.

```csx
// HierarchyPanel.csx:244 - renders ALL entities
foreach (var root in roots) AddRows(root, 0);

// InspectorPanel.csx:178 - renders ALL components
foreach (var comp in entity.Components.ToList()) {
```

**Impact**: Performance degrades significantly with 100+ entities/components.

**Fix**: Use Paper's `UI.VirtualList` for large lists.

### 2. AlwaysRender = true (Program.cs:26)

```csharp
surface.AlwaysRender = true;
```

**Issue**: Editor re-renders every frame even when idle. Paper's diffing should handle updates, but `AlwaysRender = true` bypasses this optimization.

**Impact**: Unnecessary CPU usage.

**Fix**: Set to `false` and use `host.MarkDirty()` to trigger re-renders only when needed.

---

## MEDIUM Priority Issues

### 3. Direct State Mutation (InspectorPanel.csx, ScenePanel.csx)

**Issue**: Mutating entity properties directly instead of through controlled updates.

```csx
// InspectorPanel.csx:103
onChange: v => { entity.Name = v; host.MarkSceneDirty(); }

// ScenePanel.csx:200
host.SelectedEntity.Transform.Position = new ...;
```

**Impact**: Bypasses Paper's reconciliation - may cause stale renders or missed updates. The `MarkDirty()` calls mitigate this but it's not idiomatic Paper.

**Fix**: Consider using `UseState` for transform values and syncing on commit.

### 4. No Memoization of Expensive Operations (InspectorPanel.csx)

**Issue**: `EditorReflection.EditableProperties()` and `EditorReflection.Classify()` called every render.

```csx
// InspectorPanel.csx:183 - called every render
foreach (var pi in EditorReflection.EditableProperties(compType).Take(16))
```

**Fix**: Cache component property lists in EditorGameHost or use `UseMemo`.

### 5. Multiple UseStable Hooks for State (Various panels)

**Issue**: Using `Hooks.UseStable` with arrays for state management - works but not idiomatic.

```csx
var renamingId = Hooks.UseStable(() => new string[] { "" });
var renameText = Hooks.UseStable(() => new string[] { "" });
```

**Fix**: Use `Hooks.UseState` for better Paper integration and clarity.

---

## LOW Priority Issues

### 6. Missing Key Props in Some Mappings

**Issue**: Some list mappings may benefit from explicit keys for better diffing.

### 7. Drag State Complexity (ScenePanel.csx)

**Issue**: Using float array for drag state is fragile.

```csharp
var dragState = Hooks.UseStable(() => new float[9]);
```

**Fix**: Use a simple record/class for drag state.

### 8. Console Panel Not Using VirtualList

**Issue**: ConsolePanel likely renders all log entries.

---

## Recommendations

1. Add VirtualList support for Hierarchy, Inspector components, Asset Browser
2. Remove `AlwaysRender = true` and rely on dirty flags
3. Add memoization for reflection results
4. Consider using UseState more consistently over UseStable for UI state
